## Context

- 后端 `DELETE /tasks/{id}` 已实现软删除：写入 `deleted_at`、移除向量，并在任务查询时默认过滤已删除记录。
- 当前缺口位于用户入口层：CLI 无删除命令，Web 任务页面无删除操作，导致现有能力无法通过标准工作流触达。
- 删除属于高风险动作，需要最小化误删风险，同时保证删除后的界面与任务列表状态立即同步。

## Goals / Non-Goals

- Goals:
  - 为 CLI 和 Web 提供稳定、可确认的删除入口
  - 复用后端现有权限与软删除机制
  - 删除后让列表、详情、看板等任务视图同步刷新
  - 明确失败提示和验证路径
- Non-Goals:
  - 不改造后端删除为硬删除
  - 不在本次任务中新增“恢复任务”入口
  - 不自动清理 Git 分支或 worktree

## Decisions

- Decision: CLI 删除采用显式 `ctx tasks delete <task_id>` 形式
  - Why: 删除是高风险操作，显式传入任务 ID 比“默认删除当前分支绑定任务”更安全，也更适用于非当前分支任务的管理。

- Decision: Web 删除入口优先落在任务详情页和任务列表页
  - Why: 详情页语义最清晰，列表页能覆盖高频管理路径；不在看板卡片上首轮增加删除按钮，可避免把高风险动作塞进拖拽密集区域。

- Decision: 删除成功后统一失效任务相关查询缓存，而不是仅做局部乐观移除
  - Why: 删除会影响任务详情、我的任务列表、项目任务列表、看板数据和统计展示，统一失效比手写多处乐观更新更稳妥、回滚成本更低。

## Alternatives Considered

- Alternative: CLI 删除默认推导当前分支任务 ID
  - Rejected: 误删风险更高，且对不在目标分支上的任务无帮助。

- Alternative: Web 仅在详情页提供删除入口
  - Rejected: 虽然改动更小，但任务列表仍缺少高频管理入口，不足以体现“Web 端支持删除任务”的可发现性。

- Alternative: 看板卡片也加入删除按钮
  - Rejected: 看板卡片已承担拖拽和跳转职责，继续叠加高风险动作会提高误操作概率；本次先保证详情/列表闭环。

## Risks / Trade-offs

- 误删风险
  - Mitigation: CLI 和 Web 都增加明确确认步骤；按钮采用危险态样式与文案。

- 删除后界面残留旧数据
  - Mitigation: 成功后统一失效 `myTasks`、`task/<id>`、项目任务列表和相关看板查询。

- 权限规则与用户预期不一致
  - Mitigation: 复用后端 `_ensure_task_access`，前端/CLI 只做错误提示，不在客户端复制权限规则。

## Validation Plan

- CLI:
  - 成功删除一条可访问任务
  - 删除不存在或无权限任务时给出明确错误
- Web:
  - 任务详情页删除成功后跳回列表，且列表不再显示该任务
  - 任务列表页删除成功后当前表格立即移除该任务
  - 权限或不存在错误时显示失败提示，不误更新 UI
- E2E:
  - 主路径：列表页或详情页删除任务
  - 边界路径：删除后再次打开任务详情返回不存在提示

## Open Questions

  - 暂定方案：不在前端预判权限，统一展示入口，由后端裁决并返回错误提示。
