# 业务规则配置规范

本规范定义 Cortex 项目中业务配置项的维护方式与一致性约束，避免前后端、CLI 配置不一致。

## 1. 配置目标

| 目标 | 说明 |
|------|------|
| 一致性 | 相同业务字段在前端、后端、CLI 的语义与取值保持一致 |
| 可追踪 | 配置来源明确，变更有据可查 |
| 可回归 | 配置变更后可通过固定检查项验证 |

## 2. 配置来源与职责

| 配置类型 | 主来源 | 使用方 | 备注 |
|----------|--------|--------|------|
| 任务状态/优先级显示 | 前端常量模块（`src/constants`） | 前端页面渲染 | 负责展示文案和颜色 |
| 任务状态流转 | 后端任务接口 + CLI 命令流程 | 后端、CLI | 负责状态合法流转 |
| API 路径规范 | 后端路由（`/api/v1`） | 前端、CLI | 调用方必须对齐统一前缀 |
| Git 分支约定 | CLI `tasks` 命令 | CLI、协作流程 | 与任务状态流转绑定 |
| 数据库连接配置 | 环境变量（`DATABASE_URL`） | 后端、部署脚本 | 严禁硬编码 |

## 3. 权威源声明（Single Source of Truth）

| 配置项 | Owner | 主来源 | 镜像来源 | 验收命令 |
|--------|-------|--------|----------|----------|
| `status` | 后端 API | 后端任务接口与任务 schema | 前端常量、CLI 状态命令 | `rg -n "TODO|IN_PROGRESS|REVIEW|DONE" cortex-backend cortex-frontend` |
| `priority` | 后端 API | 后端任务 schema | 前端常量、CLI 展示与输入 | `rg -n "HIGH|MEDIUM|LOW" cortex-backend cortex-frontend` |
| `type` | 后端 API | 后端任务 schema 与任务创建接口 | CLI `tasks new --type` | `rg -n "type" cortex-backend/cli cortex-backend/app` |
| API 前缀 | 后端 API | `app/main.py` 路由挂载 | 前端/CLI 请求地址配置 | `rg -n "/api/v1" cortex-backend cortex-frontend` |
| 分支命名规则 | CLI | `cli/commands/tasks.py` | Review 命令、协作规范 | `rg -n "task-\\{id\\}|task-\\$|task-" cortex-backend/cli` |

## 4. 变更流程

```bash
# 1) 先更新模块规则文档
# 2) 再修改代码配置
# 3) 执行关键验证
# 4) 在 PR 描述中记录“配置项变更清单”
```

## 5. 配置变更检查清单

| 检查项 | 说明 |
|--------|------|
| 前后端字段是否同名同义 | 如 `status`、`priority`、`type` 是否一致 |
| API 调用路径是否统一 | 是否全部走 `/api/v1` 前缀 |
| CLI 与后端状态流是否一致 | `start/pr/done` 是否映射正确后端状态 |
| 文案与颜色映射是否集中管理 | 页面中禁止重复手写映射 |
| 迁移是否同步 | 涉及数据结构变化时是否补齐 migration 与验证步骤 |

## 6. 作用范围

- 适用于：前端、后端、CLI、部署脚本中的业务配置与约定。
- 不适用于：第三方服务私有内部配置（由各服务官方规范管理）。

## 7. 严禁事项

- ⚠️ 严禁在多个模块重复定义同一业务枚举（如状态、优先级）而不维护统一来源。
- ⚠️ 严禁将敏感配置写入代码仓库（密钥、密码、Token）。
- ⚠️ 严禁先改代码后补规则文档。
